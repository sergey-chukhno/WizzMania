Architecture: The Clean Approach

"Source code dependencies must only point inward."
‚Äî Robert C. Martin (Uncle Bob)

We implemented a modular MVC-like Onion Architecture:

  • Core (Model) - The Brain
    Pure C++ logic (Grid, Tiles). Knows nothing about graphics or files.
    Role: Enforces game rules and maintains state integrity.
  • Engine (View) - The Senses
    Wraps SDL2 (Graphics, Audio, Input). Knows nothing about 2048 rules.
    Role: Handles rendering and raw I/O events.
  • Game (Controller) - The Glue
    Orchestrates the loop. Connects Inputs (Engine) to Actions (Core).
    Role: Manages Application State (Menu, Playing, GameOver).

‚úÖ Why Clean?

Testable: Core logic Verified in isolation.

Portable: Swap SDL2 for OpenGL without breaking rules.

‚ùå vs "Dirty"

Spaghetti: Game Logic calling SDL_RenderCopy.

Rigid: Impossible to unit test logic without a window.

Game Logic: The 3-Step Shuffle

Algorithm Steps

We solve 2048 using a deterministic row transformation:

  1. Compress: Remove all zeros (gaps).
  2. Merge: Combine adjacent identical values.
  3. Pad: Fill remaining space with zeros.

Complexity Analysis

Time O(W*H)
Space O(W)

Efficient single-pass processing per row.


// src/core/GameLogic.cpp
std::tuple GameLogic::slideAndMergeRow(Row& row) {
    // 1. Compress (Filter Zeros)
    std::vector buffer;
    for (auto& t : row) 
        if (!t.isEmpty()) buffer.push_back({t.getValue(), ...});

    // 2. Merge (Scan Pairwise)
    std::vector result;
    for (size_t i = 0; i < buffer.size(); ++i) {
        if (i < buffer.size()-1 && buffer[i].val == buffer[i+1].val) {
            result.push_back(Tile(buffer[i].val * 2)); // Merge
            i++; // Skip next
        } else {
            result.push_back(Tile(buffer[i].val)); // Keep
        }
    }

    // 3. Pad (Fill Zeros)
    while (result.size() < 4) result.push_back(Tile(0));
    
    return { changed, score };
}

Optimization: Monte Carlo AI

Concept: Random Sampling

Since 2048 involves randomness (spawns), we cannot use deterministic solvers like Minimax easily.

The Strategy: Simulate the future 1,000 times per move.

The Algorithm

  1. Clone current board.
  2. Move (e.g., UP).
  3. Simulate random moves until Death.
  4. Record final score.
  5. Repeat 1,000x and pick best average.

Direction getBestMove(Grid startGrid) {
    long bestTotal = -1;
    Direction bestDir = None;

    for (auto dir : moves) {
        long totalScore = 0;
        for (int i=0; i<1000; ++i) {
            Grid sim = startGrid;
            sim.move(dir);

            while (!sim.isGameOver()) {
                sim.spawnRandom();
                sim.move(pickRandomDir());
            }
            totalScore += sim.getScore();
        }
        if (totalScore > bestTotal) ...
    }
    return bestDir;
}

High Performance: Bitboards

The Need for Speed

To run 10,000 simulations per second, a std::vector<Tile> is too slow.

Solution: Pack the entire 4x4 grid into a single 64-bit integer.

  • 1 Tile = 4 Bits (Stores power of 2, e.g., '14' = 16384).
  • 1 Row = 16 Bits (4 tiles).
  • Pre-computed Moves: We don't calculate moves at runtime. We use a 65,536 entry Lookup Table.
15,000,000 Moves / Second 

// The "State" is just a number
typedef uint64_t BitBoard;

// Move Logic is an array look-up
// ROW_TABLE[65536] pre-calculates every possible row slide
BitBoard moveLeft(BitBoard board) {
    BitBoard newBoard = 0;
    
    // Process each 16-bit row
    newBoard |= (BitBoard)ROW_TABLE[(board >>  0) & 0xFFFF] <<  0;
    newBoard |= (BitBoard)ROW_TABLE[(board >> 16) & 0xFFFF] << 16;
    newBoard |= (BitBoard)ROW_TABLE[(board >> 32) & 0xFFFF] << 32;
    newBoard |= (BitBoard)ROW_TABLE[(board >> 48) & 0xFFFF] << 48;
    
    return newBoard;
}
